# Plan: Admin Panel & Smart Blog Integration for Financial Calculator App

**Goal**
Create a maintainable, secure admin interface that allows privileged users to manage the calculator components, SEO metadata, site‑wide settings, and a "smart" blog section.

---
## High‑Level Overview
1. **Authentication & Authorization** – Decide on auth method (email/password via NextAuth, OAuth providers, or both) and role‑based access control (admin vs. regular user).
2. **Admin UI** – Choose a UI library/framework (MUI, Ant Design, or Tailwind‑based custom components) and implement a dashboard layout under `/admin` route.
3. **Data Management** – Determine storage for admin‑editable data:
   - JSON/YAML files in the repo (static, requires git commits) **or**
   - A lightweight backend API using Next.js API routes with a SQLite/PlanetScale/PostgreSQL DB.
4. **Admin Capabilities** – Typical actions:
   - Enable/disable calculators.
   - Add / edit calculator definitions (title, description, SEO tags, input fields).
   - Manage SEO overrides.
   - Update site‑wide settings (theme, default language, banner messages).
   - Publish, edit, delete blog posts.
   - View analytics (optional simple page‑view counters).
5. **Smart Blog** – A blog where content can be written in Markdown, enriched with AI‑generated summaries/tags, and automatically linked to relevant calculators.
   - Store posts as Markdown files in `content/blog/` (static generation) **or** via a headless CMS (e.g., Sanity, Contentful) accessed through API.
   - Auto‑generate SEO meta (title, description, OpenGraph) using same SEO components.
   - Optionally integrate OpenAI (or similar) to generate "smart" excerpts or related‑calculator suggestions.
6. **Routing & Security** – Protect `/admin/*` routes server‑side (NextAuth `getServerSession`) and client‑side. Use `getServerSideProps`/`middleware` to redirect non‑admin users.
7. **Testing & CI** – Add unit/integration tests for admin API routes and UI components. Ensure the existing calculator tests still pass.

  - Admin UI under the same domain (`/admin`) or a separate sub‑domain (`admin.example.com`)?
- **Role granularity**:
  - Only a single `admin` role, or also `editor`/`author` roles for blog content?

---
## Proposed Implementation Steps (Tentative)
1. **Set up authentication** – Install `next-auth`, configure providers based on chosen method, add role claim to session.
2. **Create admin layout & routing** – New `pages/admin/*` with protected middleware.
3. **Implement UI components** – Dashboard, calculators list, SEO editor, site settings, blog manager.
4. **Data layer** – Build API routes (`/api/admin/*`) for CRUD operations on chosen storage.
5. **Integrate blog** – Add `content/blog/` folder, Markdown rendering, dynamic routes (`/blog/[slug]`).
6. **Smart features** – Hook into OpenAI (or similar) to generate excerpts/tags when a post is saved.
7. **SEO integration** – Reuse existing `*Seo` components for blog posts.
8. **Testing** – Write tests for API routes, admin UI, and ensure existing calculator tests still pass.
9. **Documentation** – Update README with admin usage and blog publishing workflow.
